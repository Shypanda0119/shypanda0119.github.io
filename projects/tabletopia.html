🍽 Tabletopia — 식당 예약 & 관리 플랫폼

Spring Boot Spring MVC JPA MySQL OpenAI API React Admin

📌 프로젝트 개요

식당 예약·웨이팅·메뉴 관리·운영시간 등 식당 운영에 필요한 기능을 제공하는 서비스입니다. 저는 이 프로젝트에서 관리자 백엔드 전체 개발과 DB 기반 지역 추천 로직, 대화형 AI 기능을 통한 사용자 경험 보완을 담당했습니다.

📌 담당 역할 (My Role)

  • 관리자 페이지 백엔드 CRUD 전체 개발
  • DB 기반 지역 식당 추천 로직 직접 구현
  • OpenAI API를 활용한 대화형 메뉴 추천 기능 개발(보조 UX)
  • DB 테이블 구조 설계 참여
  • DTO 중심 조회 설계 및 N+1 발생 구간 최적화(필요 시 fetch join 고려)

📌 구현 기능 상세

① 관리자 페이지 CRUD 전체 개발

  • 식당 등록/수정/비활성화(Soft Delete)
  • 메뉴·운영시간·이미지·편의시설·리뷰 관리 API
  • React Admin에서 바로 쓸 수 있도록 JSON 응답 구조 통일
  • Soft Delete 적용 시 연관 데이터(메뉴·이미지·운영시간 등)가 함께 노출되지 않도록 조회 조건을 통일해 데이터 일관성 유지

② DB 기반 지역 추천 로직 개발

사용자가 "강남 맛집 추천"과 같이 요청하면, 지역 정보를 파싱한 뒤 DB에서 후보 식당을 조회·정렬하여 추천 결과를 반환하는 구조로 구현했습니다. (추천 대상은 활성 상태 데이터만 조회하도록 처리)

  • 입력 문장/주소에서 “OO구” 텍스트를 추출하는 파싱 로직 구현
  • 해당 지역 식당만 필터링하는 1차 추천 로직 직접 구현
  • 카테고리·평점 등의 기준으로 후보 정렬 및 상위 5개 추천

③ ChatGPT 기반 대화형 추천 기능(보조 UX)

  • OpenAI Chat Completions API를 활용한 대화형 메뉴 추천 기능 구현
  • 정답이 없는 질문(오늘의 메뉴 추천 등)에 대해 사용자 경험을 보완하는 용도로 활용
  • 대화 횟수 제한(5회 이내)으로 비용/무한 대화 방지
  • API 호출 실패/예외 발생 시 기본 안내 메시지를 반환하는 fallback 처리

④ JPA 기반 조회 최적화

  • 목록/관리 화면은 DTO 조회로 필요한 값만 조회하여 성능 최적화
  • Soft Delete 데이터가 조회/추천/관리 화면에 노출되지 않도록 조회 조건 일관 적용
  • 연관 엔티티 접근으로 N+1이 발생할 수 있는 구간은 조회 전략을 분리(필요 시 fetch join 고려)
  • 이미지 등은 바이너리 대신 URL/경로 메타데이터를 DTO로 조회

⑤ DB 설계 참여

  • Restaurant, Menu, OpeningHour, Image, Facility, Review 테이블 구조 설계 참여
  • 운영/추적을 고려해 Soft Delete(예: deleted flag, deletedAt) 기반으로 데이터 상태 관리

📌 기술적 문제 & 해결 과정

  • 운영시간 복수 등록/수정 시 데이터 일관성을 위해 트랜잭션 단위로 처리
  • OpenAI API 호출 실패/응답 지연 등의 상황을 고려해 fallback 응답을 구성하여 기능이 끊기지 않게 처리
  • Soft Delete 적용 후에도 조회/추천 결과에 비활성 데이터가 섞이지 않도록 공통 조회 조건을 정리

📌 결과

관리자 기능을 통해 식당 운영 데이터를 안정적으로 관리할 수 있도록 했고, DB 기반 지역 추천 로직과 대화형 AI(보조 UX) 기능을 결합해 실제로 작동하는 레스토랑 관리·추천 플랫폼 백엔드를 구축했습니다. 또한 Soft Delete 기반으로 운영 데이터를 추적 가능하게 유지하면서, 비활성 데이터가 노출되지 않도록 조회 조건을 통일했습니다.
