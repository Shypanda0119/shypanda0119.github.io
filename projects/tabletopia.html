Tabletopia

식당 예약, 운영 관리, 지역 기반 추천 기능을 제공하는 레스토랑 플랫폼 프로젝트입니다.

Spring Boot Spring MVC JPA MySQL OpenAI API React Admin

프로젝트 소개

식당 예약, 웨이팅, 메뉴 관리, 운영시간 관리 등 식당 운영에 필요한 기능을 제공하는 서비스입니다. 관리자 기능과 사용자 추천 기능을 함께 구현하며 운영과 추천 흐름이 연결되는 백엔드 구조를 설계했습니다.

담당 역할

  • 관리자 페이지 백엔드 CRUD API 개발
  • 지역 기반 식당 추천 로직 구현
  • OpenAI API 기반 대화형 추천 기능 구현
  • DB 테이블 구조 설계 참여
  • DTO 조회 구조 및 성능 최적화 적용

핵심 구현 기능

1. 관리자 운영 기능 구현

식당 등록, 수정, 비활성화와 함께 메뉴, 운영시간, 이미지, 편의시설, 리뷰를 관리할 수 있는 관리자 API를 구현했습니다. React Admin에서 바로 사용할 수 있도록 응답 구조를 일관되게 맞췄습니다.

2. 지역 기반 식당 추천 기능 구현

사용자의 입력 문장에서 지역 정보를 추출한 뒤, 해당 지역의 식당만 조회하여 추천하는 기능을 구현했습니다. 카테고리와 평점 기준으로 후보를 정렬해 상위 5개 식당을 추천하도록 구성했습니다.

3. OpenAI API 기반 대화형 추천 기능 구현

정답이 정해져 있지 않은 질문에 대응할 수 있도록 대화형 메뉴 추천 기능을 구현했습니다. 대화 횟수를 제한하고, API 호출 실패 시 fallback 응답을 제공해 사용자 경험이 끊기지 않도록 했습니다.

4. 조회 최적화 및 데이터 일관성 유지

목록과 관리 화면은 DTO 조회 방식으로 필요한 값만 조회해 성능을 최적화했습니다. 또한 Soft Delete가 적용된 데이터가 조회, 추천, 관리 화면에 노출되지 않도록 공통 조회 조건을 정리해 데이터 일관성을 유지했습니다.

기술적 문제와 해결

1. 운영시간 다중 수정 시 일부 데이터만 반영될 수 있는 문제

운영시간은 여러 건을 한 번에 등록하거나 수정하는 구조였기 때문에, 중간에 예외가 발생하면 일부 데이터만 반영되는 문제가 생길 수 있었습니다.

이를 해결하기 위해 관련 작업을 하나의 트랜잭션으로 묶어 모든 작업이 성공할 때만 데이터가 반영되도록 처리했습니다.

2. OpenAI API 실패 시 추천 기능이 중단되는 문제

외부 API에 의존하는 구조상 응답 지연이나 실패가 발생하면 대화형 추천 기능이 완전히 끊길 수 있었습니다.

이를 해결하기 위해 기본 안내 메시지를 반환하는 fallback 응답 구조를 적용해 기능이 중단되지 않도록 했습니다.

3. Soft Delete 데이터가 조회 결과에 섞이는 문제

식당, 메뉴, 운영시간 등 비활성화된 데이터가 조회와 추천 결과에 함께 노출될 가능성이 있었습니다.

이를 해결하기 위해 공통 조회 조건을 정리해 비활성 데이터가 화면에 노출되지 않도록 일관성을 맞췄습니다.

결과

관리자 기능을 통해 식당 운영 데이터를 안정적으로 관리할 수 있는 구조를 만들었고, 지역 기반 추천 로직과 대화형 추천 기능을 결합해 실제로 동작하는 레스토랑 관리·추천 플랫폼 백엔드를 구현했습니다. 또한 Soft Delete 기반 데이터 관리와 조회 조건 통일을 통해 운영 데이터의 추적 가능성과 화면 노출 일관성을 함께 확보했습니다.
